<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Expo client -> Supabase anon client | Screen input and auth state determine which salesperson records are fetched or written. |
| Auth identity -> sales_persons lookup | The profile screen must only merge the current user with the matching salesperson record. |
| Device location -> attendance write | GPS data is untrusted and can be absent, stale, or spoofed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | I | app/(tabs)/index.tsx | mitigate | Query only the columns needed for the dashboard and keep reads limited to the sales tables listed in phase 1. |
| T-01-05 | T | app/(tabs)/attendance.tsx | mitigate | Validate and normalize coordinates, timestamps, and optional fields before insert/update so malformed location data cannot poison the attendance history. |
| T-01-06 | S | app/(tabs)/profile.tsx | mitigate | Resolve the current user first, then fetch only the matching sales_persons row so the screen cannot impersonate another salesperson. |
</threat_model>
